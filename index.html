▼ The runtime brain

The Master-Journey-Aware Retriever

It's the Phase 9 retriever, generalised. Instead of returning a subgraph from one module, it walks across module boundaries when the user's question crosses them — assembling a packet that contains nodes from each module it touched and the lifecycle edges between them. The LLM then writes one coherent answer that follows the chain.

Step 1 · the question

User asks a multi-module question

Real users don't think in modules. They ask things that span the chain.

"My PR was approved last week, where's the PO? And did the goods arrive yet?"
Step 2 · master + modules

Classify, then identify touched modules

The retriever first classifies the master journey, then enumerates every module the question references.

Master:  P2P

Modules touched:
PR PO GR

Step 3 · assemble subgraph

Walk the cross-module edges

Pull the relevant nodes from each module, then add the lifecycle edges that connect them.

PR.Approved ──▶ PO.Draft
PO.Authorized ──▶ GR.Created

Resulting packet → ~12 graph nodes (across 3 modules) + 2 cross-module edges + evidence summaries. Compact, focused, multi-module.

1 · The Problem

A hundred clicks, or one sentence.

Every PO in Ramco today is a click-through across 14 sub-screens and 200+ fields, with conditional rules that depend on what the user just picked. We want a buyer who's never opened the ERP to say "I need 50 pumps from Acme" and get a complete PO.

Before · Screen-driven

User opens "Create Direct PO" screen. Picks Type from dropdown. Realises Capital needs a Proposal — opens another screen. Comes back. Picks Folder, Buyer, Currency, Tax Region… every field one of hundreds.

PoCrtMain → cbopotype → cbobuyer
→ Specify Dropship Address (12 fields)
→ Specify Budget Details (8 fields)
→ Specify Terms (27 fields)
…14 sub-screens. 200+ controls. Total.
After · Chat-driven

User says it once, in their own words. The assistant infers which fields were supplied, which need asking, and only asks the ones that are both required and ambiguous. The conversation is about the order — not the software.

"50 industrial pumps from Acme by next month,
dropship to our Mumbai client"

bot: "Got it — dropship for 50 pumps. Currency?"

…5 turns later: PO/2026/38863 in Draft.

4 · The 13 PO Journeys

Entry-point disjoint. Lifecycle linked.

Every PO question routes to exactly one of these 13 journeys — the intent router never picks two. But once you're inside a journey, they're deeply connected: shared statuses, shared slots, shared rules, and a clear sequence of who hands off to whom. Here's the whole map.

"Are the 13 journeys linked, or independent?" — both, in two specific ways.
Entry-point disjoint

When a user types their first message, exactly one journey matches. If two could, the bot asks one clarifying question rather than guessing. Overlapping entry points would mean the same message routes to two different conversations — we'd never recover. This is enforced by the splice route_create_disambiguation.

Lifecycle linked

A PO created in one journey is approved in another, amended in a third, closed in a fourth. They share the PO entity, its status state machine, and many slots / splices / rules. Twelve isolated journeys would mean re-defining "PO Type" twelve times — the graph keeps them DRY.
